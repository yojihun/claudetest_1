# 1. 프로젝트 선언

이 프로젝트는 보건의료정보관리사 국가시험을 준비하는 학습자가 개념을 이해하고, 문제를 풀고, 실무 맥락에 적용하며, 반복 학습을 통해 합격 수준에 도달하도록 돕는 통합 학습 플랫폼을 만든다.

최종 결과물은 단일한 책이 아니다. 모든 콘텐츠를 구조화된 Markdown으로 관리하고, 동일한 원본에서 다음 결과물을 생성한다.

- GitHub로 버전 관리하고 원격 저장소에 push한다.
- Vercel로 웹 앱을 배포하고 운영한다.
- 반응형 웹 앱
- 모바일 앱 또는 PWA
- 학습 카드와 플래시카드
- OX·객관식·단답형·사례형 문제
- 모의고사와 오답노트
- PDF 및 DOCX 교재
- 강사용 자료
- 향후 AI 튜터용 지식베이스

# 2. 프로젝트 목표

## 2.1 교육 목표

학습자는 다음을 할 수 있어야 한다.

1. 보건의료정보관리의 핵심 개념을 정확히 설명한다.
2. 의료기록, 질병분류, 의료행위 분류, 병원통계, 암등록 및 관련 법규를 통합적으로 이해한다.
3. 국가시험의 지식형·판단형·계산형·사례형 문제를 해결한다.
4. 실제 의료기관의 정보 흐름과 보건의료정보관리사의 업무를 설명한다.
5. 개인정보 보호, 데이터 품질, 의료정보 표준 및 디지털 헬스 환경의 변화를 이해한다.
6. 틀린 문제의 원인을 분석하고 취약 개념을 반복 학습한다.

## 2.2 제품 목표

- 학습 내용과 문제를 작은 단위로 구조화한다.
- 학습자의 진도와 정답 이력을 저장한다.
- 학습자의 오답 및 다시 보고 싶은 문제를 오답노트로 저장한다.
- 중요도, 출제 가능성, 난이도, 선수 지식에 따라 학습 경로를 제시한다.
- 법령과 표준의 기준일을 표시한다.
- 모든 문제를 근거가 되는 Topic 및 출처와 연결한다.
- 앱 콘텐츠를 책으로 내보낼 수 있게 한다.

# 3. 대상 사용자

## 3.1 1차 사용자

- 보건의료정보관리사 국가시험 수험생
- 관련 학과 재학생 및 졸업예정자
- 재응시자

## 3.2 2차 사용자

- 보건의료정보관리 관련 학과 교수자
- 의료기관 신규 보건의료정보관리사
- 의무기록, 분류, 통계, 암등록 업무 입문자

# 4. 품질 기준

콘텐츠는 다음 조건을 모두 충족해야 한다.

- **정확성:** 공식 자료와 전문 문헌에 근거한다.
- **완전성:** 시험 범위의 핵심 개념을 누락하지 않는다.
- **명확성:** 한 문장에 하나의 핵심 내용을 담는다.
- **일관성:** 동일 용어와 약어를 동일한 방식으로 표기한다.
- **추적성:** 주장, 문제, 해설이 출처와 연결된다.
- **최신성:** 법령·시험제도·분류체계의 기준일을 기록한다.
- **실용성:** 단순 정의를 넘어 병원 업무와 연결한다.
- **평가 가능성:** 각 Topic마다 학습목표와 점검문항이 있다.
- **접근성:** 색상만으로 정보를 구분하지 않고 대체 텍스트를 제공한다.

# 5. 핵심 설계 원칙

## 5.1 Single Source of Truth

Markdown 콘텐츠가 유일한 원본이다. 앱 화면, PDF, DOCX, 문제은행, 플래시카드는 원본 콘텐츠를 변환해 생성한다.

## 5.2 Topic First

책의 페이지나 장이 아니라 하나의 독립된 개념인 `Topic`을 기본 단위로 한다.

좋은 Topic의 예:

- EMR과 EHR의 차이
- 완전성과 정확성
- POMR과 SOAP
- 병상이용률
- 주진단 선정
- 암등록의 원발부위

너무 큰 Topic의 예:

- 보건의료정보관리 전체
- 의료법 전체
- 질병분류 전체

## 5.3 Evidence First

법령, 시험제도, 공식 분류체계, 청구기준 및 의료정보 표준은 기억이나 2차 요약에만 의존하지 않는다. 공식 문서의 기준일, 버전, URL 또는 문서 식별자를 저장한다.

## 5.4 Learning Before Testing

문제 수를 늘리는 것보다 개념 구조와 해설의 질을 우선한다. 모든 문제는 정답뿐 아니라 오답이 틀린 이유를 설명해야 한다.

## 5.5 App and Book Compatibility

앱에서는 Topic 단위로 보여주되, 출판 시 Volume–Chapter–Section 순서로 병합할 수 있어야 한다.

앱은 다음 모드를 지원한다.

- 핵심 암기

- 흐름도 학습

- OX 퀴즈

- 계산 연습

- 모의고사

## 6.1 기본 사용자 흐름

사용자가 앱에 접속하면 복잡한 설정 없이 바로 학습을 시작할 수 있어야 한다.

1. 첫 화면에서 오늘의 학습 콘텐츠 1개와 관련 문제 2~3개를 본다.
2. 학습 콘텐츠를 읽고 문제를 푼다.
3. 문제 제출 후 정답 여부, 해설, 문제 해결에 필요한 학습 내용을 다시 확인한다.
4. 틀렸거나 나중에 다시 보고 싶은 문제는 사용자가 오답노트에 저장한다.
5. 오답노트에 저장한 문제는 이후 다시 풀기 흐름에서 재노출한다.

## 6.2 매일 학습 규칙

- 첫 화면은 `매일 학습`이다.
- 접속할 때마다 Topic과 문제 2~3개를 무작위로 다시 구성한다.
- 무작위 선택은 완전 균등이 아니라 사용자의 집중학습 설정을 반영한 가중 무작위 방식으로 한다.
- 학습 묶음은 가능한 한 같은 Topic 또는 강하게 연결된 Topic을 중심으로 구성한다.

## 6.3 집중학습 설정

- 사용자는 설정 화면에서 집중학습 대상을 `Volume` 또는 `Chapter` 단위로 선택할 수 있다.
- 집중학습 대상으로 지정된 범위의 Topic과 문제는 `매일 학습` 및 랜덤 문제 제시에서 더 자주 노출된다.
- 집중학습 설정은 사용자 기기에 저장하고, 로그인 기능 도입 후에는 계정 단위로 확장할 수 있게 설계한다.

## 6.4 앱 메뉴 구조

앱의 1차 메뉴는 다음 네 가지로 고정한다.

1. `매일 학습`
2. `단원별 학습`
3. `의학용어 사전`
4. `설정`

`매일 학습`은 랜덤 학습 묶음을, `단원별 학습`은 Volume/Chapter 중심 탐색을, `의학용어 사전`은 용어 검색과 정의 확인을, `설정`은 집중학습과 개인 학습 옵션을 담당한다.

## 6.5 배포 자동화

- GitHub `main` 브랜치를 배포 기준 브랜치로 사용한다.
- Vercel 프로젝트는 GitHub 원격 저장소와 연결하고, `main`에 push되면 자동으로 Production 배포가 실행되게 한다.
- Preview 배포는 pull request 또는 브랜치 push 기준으로 자동 생성되게 한다.
- 앱 빌드 시 `apps/web/scripts/generate-content.mjs`를 자동 실행해 최신 `content`, `questions`, `GLOSSARY.md` 원본을 다시 읽고 `src/generated/learning-dataset.json`을 재생성한다.
- 따라서 원본 markdown/yaml 수정 후에는 생성 파일을 수동으로 갱신하지 않아도 되며, GitHub push만으로 최신 콘텐츠가 배포되어야 한다.
- 단, Git에 반영되지 않은 로컬 파일은 자동 배포 대상이 아니므로 반드시 원본 파일을 commit/push해야 한다.

# 8. 데이터 원칙

- 모든 Topic에는 고유 ID를 부여한다.
- ID는 한번 배포한 후 변경하지 않는다.
- 제목 변경과 ID 변경을 분리한다.
- 문제는 연결된 Topic ID를 반드시 가진다.
- 이미지와 표는 연결된 Topic ID를 가진다.
- 법령 관련 콘텐츠는 `as_of_date`를 가진다.
- 미검증 정보는 `verified: false`로 표시한다.
- 폐기된 콘텐츠는 삭제보다 `status: deprecated`를 사용한다.

# 10. AI 작업 원칙

AI가 콘텐츠를 작성하거나 수정할 때 다음을 지킨다.

- 필요할 때만 Gemini API 등 외부 생성형 AI API를 사용하며, API Key는 환경변수로 주입한다.
- 생성형 AI 사용 여부와 적용 범위를 앱 기능 단위로 분리한다.
- 공식 자료가 필요한 사실을 추측하지 않는다.
- 최신성 확인이 필요한 내용에는 기준일을 표시한다.
- 존재하지 않는 법 조항, 통계, 출처를 만들지 않는다.
- 불확실한 내용은 명확히 표시한다.
- 원문을 과도하게 복제하지 않고 요약·재구성한다.
- 문제의 정답은 근거와 함께 검증한다.
- 동일 개념의 정의가 파일마다 달라지지 않도록 GLOSSARY를 확인한다.
- 대규모 수정 전 관련 Topic과 문제의 연결 관계를 확인한다.
- 사용자에게 확인받지 않아도 되는 범위에서는 일관된 규칙에 따라 진행한다.

## 10.1 AI 기능 적용 원칙

- 초기 MVP는 AI 없이도 완전한 학습 흐름이 동작해야 한다.
- Gemini API 또는 다른 외부 모델은 선택 기능으로만 붙인다.
- 우선 적용 후보는 "학습 내용 쉬운 설명", "오답 이유 재서술", "추가 연습문제 초안"이다.
- AI가 생성한 결과는 검증된 Topic, Question, Reference 데이터를 우선 근거로 사용해야 한다.
- API Key는 `.env.local` 및 배포 환경변수로만 관리하고 저장소에 커밋하지 않는다.

# 11. 금지사항

- 공식 근거 없이 시험 문항 수나 합격기준을 확정적으로 서술
- 법령 기준일 없는 법규 콘텐츠 배포
- 정답만 있고 설명이 없는 문제
- 출처와 연결되지 않은 수치 제시
- 동일한 문제를 어순만 바꾸어 반복 생산
- 학생을 속이기 위한 모호한 함정
- 개인정보가 포함된 실제 환자 사례 사용
- 저작권이 있는 표·그림의 무단 복제
- 색상만으로 정답이나 중요도를 구분
- Markdown 본문에 앱 전용 UI 코드를 무분별하게 삽입

# 12. 버전 관리

Semantic Versioning을 기본으로 한다.

- MAJOR: 데이터 구조 또는 전체 교육과정의 호환성 파괴
- MINOR: 새 Chapter, Topic, 문제 유형 또는 기능 추가
- PATCH: 오탈자 수정, UI 개선, 배포 설정 보완, 문제 해설 보정

## 12.1 개발 및 배포 원칙

- GitHub를 기본 원격 저장소로 사용한다.
- `main` 또는 배포 브랜치의 최신 안정 버전을 Vercel Production에 배포한다.
- Preview Deployment를 통해 기능 검토 후 Production으로 승격한다.
- 앱은 콘텐츠 원본 Markdown/YAML을 직접 읽거나 빌드 시 정적 데이터로 변환할 수 있어야 한다.
- 학습 기록, 오답노트, 집중학습 설정은 초기에는 로컬 저장소 기반으로 시작하고 이후 서버 저장으로 확장 가능해야 한다.
- PATCH: 오탈자, 설명 개선, 정답 오류 수정

콘텐츠 변경 이력에는 다음을 기록한다.

```yaml
change_type: correction | update | expansion | deprecation
reason:
affected_topics:
source:
reviewed_by:
```

# 13. 완료 정의

Topic은 다음 조건을 충족할 때 `published` 상태가 된다.

- 학습목표가 명확하다.
- 핵심 개념과 실무 연결이 설명되어 있다.
- 관련 용어가 GLOSSARY와 일치한다.
- 필요한 출처가 등록되어 있다.
- 최소 1회 내용 검토와 1회 문제 검토를 통과했다.
- 법령·분류·표준 관련 내용은 기준일과 버전이 있다.
- 문제의 정답과 해설이 검증되었다.
